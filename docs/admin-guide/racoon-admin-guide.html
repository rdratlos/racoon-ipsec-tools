1. Introduction

1.1 Purpose

This guide targets administrators who operate or deploy IPSec VPNs with racoon from the ipsec-tools package. The focus is on conceptual understanding of IKE protocol phases, concrete racoon configuration parameters, and known pitfalls — in particular with modern OpenSSL versions, LDAP/Samba AD authentication backends, and the Apple platforms iOS and macOS.

racoon implements IKEv1 (RFC 2409). Despite the age of the protocol it remains in use in many production environments, embedded systems, and legacy infrastructures. Understanding the RFC genealogy is the key to correct configuration.

1.2 Intended Audience

This document is written for system administrators, network engineers, and security engineers who are responsible for deploying or maintaining an IKEv1 VPN gateway based on racoon — whether for road-warrior access (Apple Cisco IPSec clients, XAuth users), site-to-site tunnels, or migration of an existing ipsec-tools installation to a modern Linux distribution. Familiarity with basic networking (NAT, routing, firewalling) and X.509/PKI concepts is assumed.

1.3 Project Scope

This repository (rdratlos/racoon-ipsec-tools) is the upstream continuation of ipsec-tools after the original SourceForge project was abandoned. It is not a fork in the sense of a divergent branch of an active project; the full commit history from the KAME origin through the SourceForge releases (up to and including 0.8.2) is preserved, and changes tracked in NetBSD's CVS repository, Apple's Darwin/macOS fork, and FreeBSD's ports tree are reviewed and selectively integrated.

The primary goal is to keep racoon buildable and functional on modern Linux distributions and with current OpenSSL releases, without changing the protocol scope or the fundamental architecture of the original code.

1.4 When to Use This Project

racoon/ipsec-tools (this continuation) is the right choice when:

  • You must interoperate with Apple macOS or iOS clients using the built-in "Cisco IPSec" VPN type.
  • You are maintaining or migrating an existing racoon-based deployment on a distribution that has dropped the package.
  • Your counterpart is a legacy device or embedded system that speaks IKEv1 only.
  • You need a minimal-footprint IKE daemon without plugin infrastructure.
  • You need per-user IP address assignment for roadwarriors driven from a directory (OpenLDAP or Samba AD DC), combined with certificate-based Phase 1 authentication — see Section 6.

You should probably choose strongSwan or Libreswan instead when:

  • All your VPN peers can speak IKEv2.
  • You need MOBIKE for mobile client roaming between networks.
  • You need EAP authentication for roadwarriors without certificates.
  • You are starting a new deployment with no legacy constraints.

IKE Protocol Overview

IKE is a hybrid protocol based on two underlying security protocols: the Internet Security Association and Key Management Protocol (ISAKMP) and the OAKLEY Key Determination Protocol (OAKLEY). ISAKMP provides a framework for authentication and key exchange but does not define them. OAKLEY describes a series of key exchanges called "modes" and details the services provided by each.

IKE runs on UDP port 500 (and 4500 for NAT-T) and is divided into two phases:

A Security Association (SA) is always unidirectional: bidirectional communication requires two SAs. Each SA is identified by a Security Parameter Index (SPI) carried in the ESP/AH header.

The Security Policy Database (SPD) (configured via setkey) determines which traffic must be protected by IPSec. The Security Association Database (SAD) holds active SAs with their key material.

ESP and AH Protocols

Two security protocols process traffic between peers once SAs are established:

racoon operates exclusively in Tunnel mode for client VPN scenarios. Transport mode is not supported for VPN client connectivity.

2.3 Main Mode (Identity Protect Mode) — RFC 2409, §5.4

Main Mode uses six messages (three exchanges) and delays transmission of peer identities until key material is available to encrypt the remaining packets. This prevents identity values from being read by third parties but places restrictions on the identity types usable with pre-shared key authentication: the address identity type is the only type that can be used with main mode + PSK, because the peer's FQDN identity cannot be used to look up the PSK before a packet is received. 

# Main Mode message flow
Initiator               Responder
─────────────────────────────────────────
HDR, SA               →               # Crypto proposal
                      ← HDR, SA        # Selection
HDR, KE, Ni           →               # DH public key, nonce
                      ← HDR, KE, Nr    # DH public key, nonce
HDR*, IDii, HASH_I    →               # Identity (encrypted)
                      ← HDR*, IDir, HASH_R

2.4 Aggressive Mode — RFC 2409, §5.5

Aggressive Mode compresses the negotiation into three messages. Peer identities are transmitted in the clear before key material is available. It is required for road-warrior scenarios with dynamically addressed clients (iOS/macOS Cisco IPSec, XAuth). Due to its payload structure, the hash algorithm and DH group cannot be auto-negotiated — they must be identically configured on both sides.

Aggressive Mode exposes the peer identity (IDii) unencrypted and allows offline dictionary attacks on pre-shared keys. Certificate-based authentication is strongly preferred when using Aggressive Mode. 
# Aggressive Mode message flow
Initiator                        Responder
─────────────────────────────────────────────────
HDR, SA, KE, Ni, IDii           →               # Identity in clear
                                 ← HDR, SA, KE, Nr, IDir, HASH_R
HDR*, HASH_I                    →

2.5 Quick Mode (Phase 2) — RFC 2409, §5.5

Quick Mode is used exclusively for Phase 2 to negotiate IPSec SAs. All packets are encrypted under the previously established ISAKMP SA. See Quick Mode message flow below for details.

Informational Mode

Used to transmit Notification or SA Deletion messages between peers. Whenever possible, informational exchanges are encrypted using an established ISAKMP SA. Unlike other exchange types, informational exchanges are unidirectional.

2.6 NAT Traversal

The Problem

ESP packets carry no port information and are delivered directly over IP (Protocol 50). NAT devices work by mapping source ports — they cannot correctly handle ESP packets. AH packets are even more problematic because AH authenticates the IP header including the source IP address, which NAT changes.

NAT-T Mechanism (RFC 3947 / RFC 3948)

NAT-T encapsulates ESP packets in UDP datagrams (port 4500). Detection happens during Phase 1 via NAT-Detection (NAT-D) payloads: both peers hash their IP/port combinations; if the hashes differ, NAT is present. 

# NAT-T detection in Phase 1 (schematic)
Initiator                              Responder
────────────────────────────────────────────────
HDR, SA [+ NAT-D vendor ID]    →
                               ← HDR, SA [+ NAT-D vendor ID]
HDR, KE, Ni, NAT-D, NAT-D     →       # NAT-Detection payloads
                               ← HDR, KE, Nr, NAT-D, NAT-D
# If NAT detected: switch to UDP port 4500
HDR*, IDii, HASH_I             →       # via UDP 4500
                               ← HDR*, IDir, HASH_R
racoon negotiates NAT-T automatically when nat_traversal on is set. Both draft (v00–v03) and RFC versions of the NAT-T extension are supported, ensuring compatibility with a wide range of client implementations.

3.3 OpenSSL 3.x

This continuation links against OpenSSL 3.x via a compatibility shim (openssl_compat.h/c) that maps legacy EVP calls used throughout the original codebase onto the OpenSSL 3.x EVP API, while remaining buildable against OpenSSL 1.1.x and 1.0.2. No special configure flags are required for OpenSSL 3.x support — it is auto-detected at ./configure time. See Section 10.2 for migration notes on PKCS#12 changes.

3.4 LDAP Support

LDAP support, used for XAUTH authentication and per-user Mode Config address assignment (Section 6), is enabled at compile time with: 

./configure --with-libldap

3.5 RADIUS Support

The --with-libradius configure option enables a RADIUS backend for XAUTH authentication (auth_source radius), backed by FreeBSD's libradius. This was a functional feature in the original ipsec-tools codebase.

This backend is not maintained in this continuation project: libradius is not packaged in current Debian or Ubuntu, so building against it requires vendoring the library and adding an unreviewed dependency. The XAUTH credential validation use case is better served by the native LDAP path against a modern directory (Samba AD, OpenLDAP), which additionally provides Mode Config address assignment. Deployments requiring RADIUS for roadwarrior authentication should use strongSwan with EAP-RADIUS over IKEv2 instead, which provides proper RADIUS accounting, CoA support, and active security maintenance.

The --with-libradius option is retained so that existing build scripts do not break, but it is not tested and is not recommended for new deployments.

3.6.2 Debian / Ubuntu Build

On Debian and Ubuntu, always pass --with-kernel-headers=/usr/include explicitly. Omitting this flag causes configure to search /usr/src/linux/ which is typically absent on these distributions, resulting in missing PF_KEY header errors during compilation (linux/pfkeyv2.h, linux/ipsec.h). The linux-headers-generic package installs these headers under /usr/include/linux/, so /usr/include is the correct search root.

  --enable-frag \

4. Configuration Basics

4.1 racoon.conf

racoon.conf: listen Block

The listen block configures the network interfaces and protocols that racoon will use to communicate. If no listen directive is specified, racoon listens on all available interface addresses. 

#
# Listen Section
#
listen
{
    # Bind IKE to specific address:port
    isakmp      10.1.1.1 [500];

    # NAT-T socket — required for clients behind NAT
    isakmp_natt 10.1.1.1 [4500];

    # Optional: require all ISAKMP addresses to be bound
    # strict_address;
}

listen Reference

isakmp_natt address [port]; Same as isakmp but also sets socket options to accept UDP-encapsulated ESP for NAT-Traversal. Must define at least one with port 4500. No default — required for NAT-T clients.
strict_address; Require all ISAKMP addresses to be bound. Ignored if no address is specified.

The timer block configures event timers used by racoon. The most important setting for NAT-T deployments is the keepalive interval.

racoon.conf: remote Block (Phase 1)

The remote block configures Phase 1 parameters for a given peer or peer group. racoon first attempts to find a remote block matching the peer's address; if none matches, the anonymous block is used. Use a specific address for site-to-site VPNs and anonymous for road warrior / VPN client scenarios where the client IP is unknown.

PSK + XAuth Remote Block Example

remote anonymous
{
    exchange_mode        aggressive;
    verify_identifier    on;
    my_identifier        fqdn "vpngw.example.com";
    peers_identifier     fqdn "client.example.com";

    passive              on;
    generate_policy      unique;
    ike_frag             on;
    nat_traversal        on;
    dpd_delay            30;
    proposal_check       claim;
    lifetime             time 24 hours;

    proposal {
        encryption_algorithm    aes 256;
        hash_algorithm          sha1;
        authentication_method   xauth_psk_server;
        dh_group                14;
    }
}

RSA + XAuth Remote Block Example

remote anonymous
{
    exchange_mode        main;
    verify_identifier    on;
    my_identifier        asn1dn;
    peers_identifier     asn1dn;

    # When asn1dn is used without a value, racoon uses
    # the certificate Subject DN as identifier.
    certificate_type     x509 "vpngw.crt" "vpngw.key";
    ca_type              x509 "ca.crt";

    passive              on;
    generate_policy      unique;
    ike_frag             on;
    nat_traversal        on;
    dpd_delay            30;
    proposal_check       claim;
    lifetime             time 24 hours;

    proposal {
        encryption_algorithm    aes 256;
        hash_algorithm          sha256;
        authentication_method   xauth_rsa_server;
        dh_group                14;
    }
}

4.2 setkey

setkey provides direct, low-level access to the kernel's XFRM Security Policy Database (SPD) and Security Association Database (SAD), independent of racoon. It is useful for manual SA configuration, scripting, and environments where IKE negotiation is not needed at all. 

# Show active Security Associations (SAD)
setkey -D

# Show Security Policy Database (SPD)
setkey -DP

# Flush all SAs / policies
setkey -F
setkey -FP

# Load a setkey script
setkey -f /etc/ipsec-tools.conf

4.3 Security Policies

The Security Policy Database (SPD) determines which traffic must be protected by IPsec, and with what mode/protocol. racoon populates the SPD indirectly via generate_policy (responder side, see remote block) or policies can be defined statically with setkey. 

# Inspect policy database
setkey -DP
ip xfrm policy

4.4 Security Associations

The Security Association Database (SAD) holds active SAs together with their key material, SPI, and lifetime. Each SA is unidirectional; bidirectional traffic requires two SAs, one per direction. 

# Inspect active SAs
setkey -D
ip xfrm state

# Per-SA statistics
ip -s xfrm state

5. Authentication Methods

5.1 Pre-Shared Keys

Both peers must share a common key value. The authenticating peer sends a Hash Value generated from the shared key and ancillary data; the responder generates the same hash locally and compares. 

# In proposal block:
authentication_method  pre_shared_key;

# In /etc/racoon/psk.txt (peer-id  secret):
client.example.com   mysharedsecret
# For address-based ID:
192.168.1.1          anothersecret
PSK with Aggressive Mode is vulnerable to offline brute-force attacks. Use long, random keys (≥ 20 characters, high entropy).

5.2 X.509 Certificates (RSA)

The peer being validated must hold a Digital Certificate signed by a trusted CA and the corresponding private key. During IKE authentication, the peer sends its certificate and a Hash Value signed with the private key. The responder verifies the certificate against the CA, then verifies the hash against the certificate's public key. 

authentication_method  rsasig;

certificate_type  x509  "gateway.crt"  "gateway.key";
ca_type           x509  "ca.crt";

Quick Mode Message Flow (Phase 2 detail)

Phase 2 (Quick Mode) negotiates the actual IPSec SAs for data transport. It runs fully encrypted over the Phase 1 IKE SA channel. One Quick Mode exchange results in two unidirectional SAs (one per direction). 

# Quick Mode message flow
Initiator                                    Responder
────────────────────────────────────────────────────────
HDR*, HASH(1), SA, Ni [, KE][, IDci, IDcr]  →
                                             ← HDR*, HASH(2), SA, Nr [, KE][, IDci, IDcr]
HDR*, HASH(3)                                →

The optional KE payload enables Perfect Forward Secrecy (PFS): a fresh DH key exchange ensures that compromise of a long-term key cannot expose past session keys.

racoon does not have a fixed list of security protocols to negotiate for Phase 2. The list is passed by the SPD (Security Policy Database) in the kernel. Therefore you must define all potential algorithms in the Phase 2 proposals even if some will never be used — racoon computes the actual proposals by permutation.

racoon.conf: sainfo Block (Phase 2)

The sainfo block defines which crypto parameters are offered for the data tunnel. Selection is based on source/destination address ranges, or anonymous for any peer. Multiple sainfo blocks can coexist. 

# Semi-anonymous: specific local subnet, any peer
sainfo subnet 10.1.2.0/24 any anonymous
{
    pfs_group                    14;
    lifetime                     time 3600 seconds;
    encryption_algorithm         aes 256, aes 128;
    authentication_algorithm     hmac_sha1;
    compression_algorithm        deflate;
}

# Fully anonymous: any subnet, any peer
sainfo anonymous
{
    pfs_group                    14;
    lifetime                     time 3600 seconds;
    encryption_algorithm         aes 256, aes 128, 3des;
    authentication_algorithm     hmac_sha1;
    compression_algorithm        none;
}

# XAuth group-restricted: only vpn-admin users may access 10.1.3.x
sainfo subnet 10.1.3.0/24 any anonymous group "vpn-admin"
{
    pfs_group                    14;
    lifetime                     time 3600 seconds;
    encryption_algorithm         aes 256;
    authentication_algorithm     hmac_sha1;
}

Perfect Forward Secrecy (PFS)

PFS ensures that compromise of a long-term private key (RSA key or PSK) does not allow decryption of previously recorded traffic. Without PFS, all past session keys could theoretically be reconstructed from the compromised long-term key.

With PFS, each Phase 2 negotiation performs a fresh Diffie-Hellman exchange. The PFS DH group and the Phase 1 DH group need not be identical — a lower group in Phase 2 speeds up frequent SA renewal. 

# PFS enabled (recommended)
sainfo anonymous {
    pfs_group 14;   # MODP-2048 for Phase 2
    ...
}

# PFS disabled (not recommended)
sainfo anonymous {
    # no pfs_group directive → no PFS
    ...
}

Mode Config (Configuration Exchange)

The Mode Config extension (ISAKMP Configuration Method, draft-ietf-ipsec-isakmp-mode-cfg) exchanges client configuration information after Phase 1 and before Phase 2. It allows the gateway to push or the client to pull: IP address, netmask, DNS, WINS, split-tunnel topology, and a login banner.

racoon supports the Pull method — the client requests settings from the gateway. This is the default and the only method supported by ipsec-tools.

When split_network include is configured, the client only tunnels packets destined for the specified networks. All other traffic uses the local connection directly. If no topology information is received from the gateway, the client falls back to tunneling all traffic.

5.3 XAUTH (Extended Authentication)

XAuth (draft-ietf-ipsec-isakmp-xauth) adds user-based authentication on top of Phase 1. After the ISAKMP SA is established using PSK or RSA, the gateway challenges the client for a username and password, which is validated against an account database.

XAuth requires generate_policy unique and passive on in the remote block. The mode_cfg block must be present and auth_source must be configured.

5.4 Hybrid Authentication

Hybrid Authentication combines one-sided RSA authentication (the gateway authenticates with a certificate; the client does not) with a subsequent XAuth user credential check. This is the model used by most commercial Cisco-compatible VPN gateways. 

authentication_method  hybrid_rsa_server;

# Gateway must have a certificate; client only needs CA cert
certificate_type  x509  "gateway.crt"  "gateway.key";
ca_type           x509  "ca.crt";

6. LDAP Authentication

6.1 Architecture

racoon supports two mutually exclusive XAUTH authentication backends, selected at compile time (Section 3.4 / 3.5) and at runtime via the auth_source directive in racoon.conf. Native LDAP, compiled in via --with-libldap, is the actively supported and recommended authentication backend for this continuation project.

racoon's native LDAP support goes beyond simple credential checking: the ldapcfg block serves a dual purpose:

#PurposeDescription
1 XAUTH authentication racoon performs an LDAP bind (or search + bind) to validate the roadwarrior's username and password during IKEv1 Phase 1.5 (XAUTH), in addition to the X.509 certificate check in Phase 1 proper. This gives genuine two-factor authentication: something you have (certificate) + something you know (directory password).
2 Mode Config address assignment racoon reads per-user IP address and netmask attributes from the user's LDAP entry and pushes them to the client during mode_cfg negotiation. This enables deterministic per-user IP assignment, essential for per-user firewall rules, audit logging, and access control policies.
The Mode Config address assignment capability is unique to the native LDAP path and cannot be replicated via PAM or RADIUS.

7.3 PKCS#12 Generation

Generating RSA Credentials

If not using PSK authentication, RSA credentials must be generated for the VPN gateway and, depending on the auth method, for clients as well. (Hybrid Authentication is the only RSA mode that does not require client certificates.) 

# Create directory structure
mkdir certs
mkdir -p demoCA/newcerts
touch demoCA/index.txt
echo "00" > demoCA/serial

# Generate CA and gateway private keys
umask 077
openssl genrsa -out certs/ca.key  4096
openssl genrsa -out certs/gw.key  4096

# Self-signed CA certificate (5 years)
umask 022
openssl req -days 1825 -x509 -new \
    -key certs/ca.key \
    -out certs/ca.crt \
    -subj "/CN=VPN-CA/O=Example"

# Gateway certificate signing request
openssl req -new \
    -key certs/gw.key \
    -out certs/gw.csr \
    -subj "/CN=vpngw.example.com/O=Example"

# Sign gateway certificate with CA
openssl ca -days 1825 \
    -in  certs/gw.csr \
    -keyfile certs/ca.key \
    -cert    certs/ca.crt \
    -out     certs/gw.crt

# Move to racoon certificate directory
cp certs/gw.crt  /etc/racoon/certs/gateway.crt
cp certs/gw.key  /etc/racoon/certs/gateway.key
cp certs/ca.crt  /etc/racoon/certs/ca.crt
chmod 600 /etc/racoon/certs/gateway.key
The CA public certificate (ca.crt) must be distributed to every client connecting to the gateway. For iOS/macOS, it must be installed as a trusted root in the device keychain.

PKCS#12 & OpenSSL 3.x

PKCS#12 files bundle certificate, private key, and optionally the CA certificate into a single encrypted file. They are typically used for import into client systems.

OpenSSL 3.x Breaking Change

OpenSSL 3.0 changed the default PKCS#12 algorithm from PBE-SHA1-3DES (traditional, supported by Apple) to AES-256-CBC with PBKDF2. This causes import failures on Apple Security.framework for older iOS/macOS versions. 

# OpenSSL 3.x: create PKCS#12 with legacy algorithms (Apple-compatible)
openssl pkcs12 -export \
    -in     client.crt \
    -inkey  client.key \
    -CAfile ca.crt \
    -chain \
    -out    client.p12 \
    -legacy \
    -keypbe  PBE-SHA1-3DES \
    -certpbe PBE-SHA1-3DES \
    -macalg  sha1

# [openssl_init]
# providers = provider_sect
# [provider_sect]
# default = default_sect
# legacy  = legacy_sect
# [default_sect]
# activate = 1
# [legacy_sect]
# activate = 1

openssl pkcs12 -info -in client.p12 -noout -legacy

7.4 iOS mobileconfig: VPN Profile & Password Lifecycle

Apple's Configuration Profile format (.mobileconfig) can deploy a complete "Cisco IPSec" VPN configuration to iOS/macOS devices — server address, XAuth/hybrid authentication, the CA certificate (or PKCS#12 identity, see 7.3), and connect-on-demand rules — without requiring the user to enter settings manually. Profiles can be signed and distributed via MDM, a web link, or email/AirDrop for manual installation.

The full com.apple.vpn.managed payload schema is documented in Apple's VPN Configuration Profile Reference. Pay particular attention to the IPSec payload's AuthenticationMethod (SharedSecret or Certificate) and XAuthEnabled keys, which must match the authentication_method configured in the racoon remote block (Section 5).
An embedded PKCS#12 identity in a mobileconfig profile is stored obfuscated, not strongly encrypted, on the device. Treat distributed profiles containing client identities as sensitive, and prefer per-user certificates over a shared one when feasible.

When distributing the "Cisco IPSec" VPN configuration to iOS devices via a .mobileconfig profile — whether sent by email, a web link, or an MDM solution — the handling of the XAuth password field has significant operational implications, particularly in environments where AD password policies enforce periodic rotation.

Password Field Behaviour

The critical decision is whether to include the XAuthPassword key in the profile payload:

Profile containsConnection behaviourPassword rotation
XAuthPassword present and non-empty iOS uses the embedded credential silently — the user is never prompted Breaks silently. VPN fails after rotation with no actionable error. Admin must re-push the profile.
XAuthPassword absent or empty string iOS prompts the user on first connection, stores the entered password in the device keychain iOS re-prompts automatically when racoon returns an authentication failure. User enters the new AD password; it is stored in the keychain and used for subsequent connections.
For any environment using AD password policies with periodic rotation, the XAuthPassword key must be omitted entirely from the mobileconfig payload. A provisioned profile cannot be edited by the user; embedding a password that will eventually expire is an operational trap.

How iOS Handles Authentication Failure (Keychain Re-prompt)

When the XAuth password stored in the device keychain is rejected by the server, the flow is:

  1. iOS initiates Phase 1 with the stored certificate credential.
  2. ISAKMP SA is established successfully.
  3. racoon sends XAuth challenge; iOS responds with the cached keychain password.
  4. racoon validates against the directory (Samba AD / LDAP) — the bind fails because the AD password has changed.
  5. racoon returns ISAKMP_NTYPE_AUTHENTICATION_FAILED.
  6. iOS interprets this specific notification as a credential rejection, discards the cached keychain entry, and presents a password prompt to the user.
  7. The user enters the new AD password. iOS stores it in the keychain and retries the connection, which succeeds.
The re-prompt mechanism depends on racoon returning the correct ISAKMP notification type on LDAP bind failure. Test this explicitly after deployment: connect once with a valid credential, change the test account's LDAP/AD password, attempt reconnection, and verify that iOS presents a password prompt rather than a generic "VPN connection failed" error. If a timeout or a generic Phase 1 failure is returned instead, iOS will not re-prompt.

MDM Considerations

If distributing profiles via an MDM server, verify that the MDM does not automatically populate XAuthPassword from a directory lookup when generating per-user profiles. Some MDM systems offer this as a convenience feature; it creates the same operational problem as embedding the password manually. The correct MDM configuration omits the password field and relies on the iOS keychain re-prompt mechanism described above.

The XAuthName field (username) may be embedded in the profile — it is not a secret and makes the initial prompt slightly more convenient since the user only needs to enter the password, not both fields. Users can also edit this field if needed; the username alone does not enable authentication without the directory password.

The XAuthName value must match the attribute configured as attr_user in ldapcfg. For Samba AD, this is sAMAccountName (the short login name, e.g. jdoe), not the UPN (jdoe@example.com) or the full DN.

7.5 Common Apple Issues

iOS: HMAC-SHA256 Pitfall in sainfo

This is one of the most common and hardest to diagnose failures in racoon–iOS connections.

The Problem

iOS expects the ESP authentication HMAC in Phase 2 to use the standard 96-bit truncation as defined in RFC 2403/2404 (HMAC-MD5/SHA1). When hmac_sha256 is configured in sainfo, iOS uses a 128-bit truncation per RFC 4868, producing an ESP Authentication Trailer that racoon cannot verify.

The result: Phase 1 and Phase 2 negotiate successfully, but no data flows — ESP packets are silently discarded because the HMAC trailer does not match.

Do not include hmac_sha256 in sainfo for iOS clients. Use only hmac_sha1. If multiple algorithms are listed, iOS selects hmac_sha256 first and then silently fails. 
# WRONG — iOS picks hmac_sha256, HMAC verification fails silently:
sainfo anonymous {
    authentication_algorithm hmac_sha256, hmac_sha1;  # ← BUG
}

# CORRECT — only hmac_sha1 for iOS compatibility:
sainfo anonymous {
    authentication_algorithm hmac_sha1;
}

8. Security Recommendations

8.1 Recommended Cipher Suites

Use AES-256 or AES-128 for encryption and SHA-256 or stronger for the Phase 1 hash/PRF (see the full tables in Phase 1 Crypto Suites). Avoid 3DES, CAST-128, Blowfish, and DES for new deployments; never use MD5.

Exception: iOS/macOS clients require hmac_sha1 (not hmac_sha256) in sainfo's authentication_algorithm due to the RFC 4868 truncation mismatch — see 7.5 Common Apple Issues.

8.2 Diffie-Hellman Groups

Use DH group 14 (MODP-2048) or higher for both Phase 1 and Phase 2 (pfs_group). Groups 1 and 2 are broken/too weak and must not be used. Full group table: Phase 1 Crypto Suites.

8.3 Aggressive Mode Considerations

Aggressive Mode with pre-shared keys is vulnerable to offline dictionary attacks and was publicly demonstrated to be broken in practice (see the 2015 Spiegel/LOGJAM research on the Diffie-Hellman parameters in use at the time). Main Mode with X.509 certificates and strong DH groups (Group 14 / 2048-bit or above) remains cryptographically adequate for most purposes, and racoon supports this configuration.

Aggressive Mode is required for road-warrior scenarios with dynamically addressed clients (iOS/macOS Cisco IPSec, XAuth), since the peer identity cannot be looked up before the PSK is applied. Where this is unavoidable, use long, high-entropy PSKs (≥ 20 characters) or switch to certificate-based authentication, and prefer hybrid/XAuth so the PSK alone does not gate access.

IKEv1 itself is a deprecated protocol from the IETF's perspective. For any new deployment where IKEv2 is available on both ends, IKEv2-based stacks (strongSwan, Libreswan) should be preferred — see Section 10.

8.4 Legacy Algorithms

Disable DES, MD5, and DH groups 1/2 wherever the peer allows it. Where a legacy peer mandates them (e.g. an old embedded VPN gateway), isolate that tunnel's remote/sainfo blocks so the weaker proposal is not offered to other, more capable peers.

Honest security assessment: racoon carries the security posture of a codebase that was last actively developed upstream around 2014. There is no active CVE triage programme, no upstream security contact, and no regular release cycle; the original SourceForge project is explicitly marked abandoned. This continuation addresses build-system and OpenSSL compatibility issues and backports fixes from Apple's and FreeBSD's forks, but it does not constitute a security-audited, production-supported release in the way that strongSwan or Libreswan releases are.

10. Migration Notes

10.1 Legacy ipsec-tools Installations

Configuration file format (racoon.conf, psk.txt, setkey scripts) is unchanged from upstream ipsec-tools; existing configurations can be reused as-is on this continuation. The main adjustments needed when migrating an old installation are packaging-related (init scripts/systemd units, certificate paths) rather than configuration-syntax changes.

10.2 OpenSSL 1.1 → 3.x

The most common breakage when moving a racoon deployment from an OpenSSL 1.1.x host to an OpenSSL 3.x host is PKCS#12 generation/import — see Section 7.3 and Section 9.5 for the legacy-provider workaround. This continuation's openssl_compat.h/c shim keeps the rest of racoon's EVP usage source-compatible across 1.0.2, 1.1.x, and 3.x.

10.3 Distribution Packaging Changes

ipsec-tools/racoon packages have been dropped from Debian Bookworm+ and Ubuntu 24.04+ repositories. This repository provides Debian and RPM packaging (Section 3.7) to fill that gap for administrators who cannot change the peer configuration on the other end of an existing tunnel.

Whether to migrate away from racoon entirely is a separate decision — see the comparison below.
